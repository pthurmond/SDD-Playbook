# 09 — Glossary and References

## Glossary

### Acceptance criteria

Specific conditions that must be true for a requirement or task to be considered complete.

### ADR

Architecture Decision Record. A short document that captures an architecture decision, alternatives considered, and consequences.

### Agent instruction file

A file such as `AGENTS.md` or `.github/copilot-instructions.md` that provides persistent guidance to AI coding agents.

### API-first development

A development approach where API contracts are designed before implementation and treated as primary artifacts.

### BDD

Behavior-Driven Development. A development practice that describes behavior using examples, often in `Given / When / Then` format.

### Business rule

A domain rule that defines how business decisions should be made.

### Clarification

The process of identifying and resolving ambiguity before planning or implementation.

### Definition of done

The conditions required before work is considered complete.

### Definition of ready

The conditions required before work is ready to enter implementation.

### EARS

Easy Approach to Requirements Syntax. A set of lightweight patterns for writing clearer requirements.

### Functional requirement

A requirement describing what the system shall do.

### Non-functional requirement

A requirement describing quality attributes such as performance, availability, accessibility, usability, security, maintainability, or observability.

### Non-goal

A clear statement of what is intentionally out of scope.

### PRD

Product Requirements Document. A document describing product intent, goals, users, scope, and requirements.

### Requirement ID

A stable identifier for a requirement, such as `FR-001` or `SEC-003`.

### SDD

Spec-Driven Development. A software delivery approach where maintained specs guide planning, implementation, testing, review, and evolution.

### Spec-anchored

An SDD maturity level where the spec is kept after implementation and maintained through future changes.

### Spec-as-source

An SDD maturity level where the spec is the primary source artifact and implementation is generated or regenerated from it.

### Spec-first

An SDD maturity level where a spec is written before implementation.

### TDD

Test-Driven Development. A practice where tests are written before implementation code.

### Traceability

The ability to connect requirements to tasks, tests, implementation, review, and release artifacts.

## Final advice

Do not start by trying to perfect the process.

Start by making the next important feature less ambiguous than the last one.

That is how SDD becomes a habit instead of another process binder that nobody uses.
